# 2025-05-08: Public surface assessment

## Decision

Assess the current public-facing structure against the question of whether it supports inhabitable reading. Finding: the surface is approximately correct; no structural changes are warranted in this phase.

Specifically refused: introducing a Reading Room / Full Archive distinction. The architecture's current unified surface — where editorial articulations, primary-source passages, and adjacency networks coexist on a single reading surface — is doing the work both modes would do. Splitting would import a vocabulary alien to the work and create maintenance burden across stewardship generations without proportional reader benefit.

## Reasoning

The phase's prompt offered the framing "Reading Room / Full Archive" as if such a distinction already existed. It does not. Engaging with whether to introduce it is itself a substantive architectural question.

Arguments against introducing the distinction:

- The current surface is already doing the work. The reading experience is unobstructed (entity bodies, passages, translations); the architectural structure is visible through adjacencies but not dominant. A reader who wants to read encounters reading; a reader who wants to perceive the architecture's structure perceives it through the existing presentation.
- A distinction would force editorial decisions about what belongs where. These decisions are themselves stewardship work that does not produce reader benefit.
- Two surfaces would mean two things to maintain across stewardship generations. Each would acquire its own conventions and drift.
- Most consequentially: the distinction would import a vocabulary alien to what the architecture has been building. ATLAS has been building one careful editorial site; calling it Reading Room and Full Archive would impose a category structure that the work has not produced.

The unified surface is correct. Future stewardship may revisit if specific cases reveal failure modes; for now there are none.

## What was assessed

- **The homepage** — threshold function, brief orientation, two regions with summaries. Appropriate.
- **Region pages** — articulation prose, entities by editorial type-order. Appropriate.
- **Entity pages** — meta line, title, subtitle, body, adjacency block. Appropriate. One small consideration (capitalization of type label) reviewed and refused as not warranted.
- **Navigation** — persistent header, adjacency-driven traversal, no search, no breadcrumb, no sibling navigation. Appropriate. Search may be added later if corpus growth makes it genuinely warranted; it is not warranted at 24 entities.
- **Visual atmosphere** — serif body, warm off-white, restrained accent, generous spacing, 38em measure. Appropriate.

## Articulated as permanent refusals

The phase explicitly articulated interface temptations the architecture must continue refusing across stewardship generations:

- Recently-added indicators
- Reading time estimates
- Reading progress indicators
- Algorithmic "related entities" sections
- Search as primary navigation
- Social annotations / public highlights
- Newsletter signups
- Comment sections
- Account-based "save" or "favorite" functions
- Trend-following visual style updates
- Mobile app versions
- AI reading assistants
- Animation or transition effects
- Continue-reading prompts
- Visitor analytics dashboards

Each will be proposed multiple times by future stewards or external advisors. The disciplines provide the articulation that should support refusals.

## What this phase did not do

No code or template changes were made. No new entities were added. No infrastructure was modified.

The phase produced explicit articulation that the public surface is approximately correct, plus explicit articulation of what should not be added. This is stewardship documentation work — the kind of work the cadence document articulates as legitimate non-production labor.

## Open questions

- When does search become genuinely warranted? Currently deferred. The cadence discipline says: when the editorial work warrants it, not before. At 24 entities, it does not.
- When does the persistent header's region list become unwieldy? With 4-5 regions it would still be acceptable; with 8+ regions it would need rethinking. The current 2-region state is fine.
- How does the public surface handle the eventual addition of new entity types (if any are added)? The current build script handles new types gracefully (they appear in editorial-then-alphabetical order at the end of the type list); the visual treatment is uniform across types. Future ontology decisions would inform any visual differentiation.
